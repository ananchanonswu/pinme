สไลด์หน้า 1: Title

สวัสดีครับ วันนี้กลุ่มของเราจะมานำเสนอโปรเจกต์ชื่อ PinMe - Location Scanner & 1-Day Trip Planner ครับ

PinMe เป็นเว็บแอปพลิเคชันสำหรับค้นหาสถานที่ใกล้เคียง และช่วยวางแผนทริป 1 วัน โดยผู้ใช้สามารถค้นหาสถานที่จากพิกัด บันทึกสถานที่โปรด และนำไปจัดเป็นแผนการเดินทางได้ครับ


สไลด์หน้า 2: Problem & Objectives

ปัญหาหลักที่เราเจอคือ ผู้ใช้มักต้องใช้เวลานานในการค้นหาและเปรียบเทียบสถานที่จากหลายแพลตฟอร์ม เช่น ต้องดูแผนที่ ระยะทาง รายละเอียดสถานที่ และรีวิวแยกกัน

อีกปัญหาคือเมื่อต้องเดินทางหลายสถานที่ภายในวันเดียว ผู้ใช้อาจจัดเวลาไม่ชัดเจน หรือเกิดเวลาทับซ้อนกัน ทำให้แผนการเดินทางไม่เป็นระบบ

ดังนั้นเป้าหมายของ PinMe คือรวมการค้นหาสถานที่ การกรองตามหมวดหมู่ การบันทึกสถานที่โปรด และการจัดตารางทริป 1 วันไว้ในเว็บเดียว เพื่อลดขั้นตอนการวางแผนและลดข้อผิดพลาดเรื่องเวลา


สไลด์หน้า 3: User Story

ในส่วน User Story เราสรุปความต้องการหลักของผู้ใช้ไว้ 9 ข้อครับ

ตัวอย่างเช่น ผู้ใช้ต้องสามารถกรอกพิกัดหรือใช้ GPS เพื่อค้นหาสถานที่ใกล้เคียงได้ สามารถเลือกรัศมีและหมวดหมู่สถานที่ได้ และสามารถดูผลลัพธ์บนแผนที่ได้

นอกจากนี้ผู้ใช้ยังสามารถดูรายละเอียดสถานที่ บันทึกสถานที่โปรด สร้างแผนทริป 1 วัน และระบบต้องสามารถตรวจสอบเวลาที่ทับซ้อนในทริปได้

ส่วนฟีเจอร์เสริมคือผู้ใช้สามารถเปลี่ยนภาษา เปลี่ยน Theme และใช้งานผ่าน 3 หน้า ได้แก่ Scanner, Favorites และ Trip Planner ครับ


สไลด์หน้า 4: Sprint Backlog

โปรเจกต์ถูกแบ่งการทำงานเป็น 4 Sprint หรือ 4 Phase หลักครับ

Sprint 1 คือ Requirement & Planning เป็นช่วงเก็บ requirement สร้าง user stories และกำหนด scope ของระบบ

Sprint 2 คือ Design & Prototype เป็นช่วงออกแบบ Prototype ด้วย Figma สร้าง Use Case Diagram และออกแบบ Architecture

Sprint 3 คือ Initial Working Product เป็นช่วงพัฒนา Frontend และ Backend ออกแบบ Data Classes หรือ Models และเริ่มทำ Unit Test

Sprint 4 คือ Final Product & Report เป็นช่วงทดสอบ UI ตั้งค่า CI Pipeline วิเคราะห์ประสิทธิภาพด้วย Profiling และจัดทำรายงานฉบับสมบูรณ์ครับ

สไลด์หน้า 5: Theme and Language

ต่อไปจะเป็นส่วนของฟีเจอร์หลักครับ

ระบบ PinMe รองรับการเปลี่ยน Theme ได้ทั้ง Light Theme และ Dark Theme และสามารถเปลี่ยนภาษาได้ระหว่างภาษาไทยกับภาษาอังกฤษ

ค่าภาษาและ Theme จะถูกบันทึกไว้ ทำให้เมื่อผู้ใช้เปลี่ยนหน้า หรือ refresh หน้าเว็บ ค่าที่เลือกไว้ยังคงอยู่ครับ


สไลด์หน้า 6: Location, Radius, Category

ในหน้า Scanner ผู้ใช้สามารถกรอกพิกัด Latitude และ Longitude หรือกดใช้ตำแหน่งปัจจุบันผ่าน GPS ได้

จากนั้นผู้ใช้สามารถกำหนดรัศมีการค้นหา เช่น 3 กิโลเมตร และเลือกหมวดหมู่สถานที่ เช่น ทั้งหมด โรงแรม ร้านอาหาร กีฬา หรือสถานที่ท่องเที่ยว

เมื่อกดปุ่มสแกน ระบบจะค้นหาสถานที่ที่อยู่ในบริเวณรัศมีที่ผู้ใช้กำหนดไว้ครับ


สไลด์หน้า 7: Results, Favorites, Trip Planner

หลังจากสแกน ระบบจะแสดงผลลัพธ์เป็นรายการสถานที่ตามหมวดหมู่ที่เลือกไว้ ผู้ใช้สามารถดูรายละเอียดของสถานที่ และกดปักหมุดหรือบันทึกเป็นสถานที่โปรดได้

นอกจากนี้ผู้ใช้สามารถเพิ่มสถานที่หรือกิจกรรมเข้าไปใน Trip Planner โดยกำหนดเวลาเริ่มต้นและเวลาสิ้นสุด ระบบจะช่วยตรวจสอบว่าเวลาทับซ้อนกับกิจกรรมอื่นหรือไม่

ปัจจุบันเว็บไซต์ถูกแยกเป็น 3 หน้า คือ Scanner, Favorites และ Trip Planner เพื่อให้ตรงกับ requirement ที่ให้จำนวนหน้าเท่ากับจำนวนสมาชิกในกลุ่มครับ


สไลด์หน้า 8: Architecture & Tech Stack

ในด้าน Architecture ระบบแบ่งเป็น Frontend และ Backend

Frontend ใช้ HTML, CSS, Vanilla JavaScript และ Leaflet.js สำหรับแสดงแผนที่ ส่วน Backend ใช้ Node.js แบบ Pure HTTP Server

Backend จะทำหน้าที่รับ request จาก Frontend แล้วเรียก SerpAPI ซึ่งใช้ Google Maps Engine เพื่อดึงข้อมูลสถานที่ ถ้า SerpAPI มีปัญหาหรือไม่มี API key ระบบจะใช้ fallback data เพื่อให้ยังสามารถทดสอบการทำงานได้

หมายเหตุ: ในสไลด์นี้ถ้ายังเขียนว่า Single Page Application ควรแก้เป็น 3-page website application เพราะตอนนี้มี 3 หน้าแล้วครับ


สไลด์หน้า 9: Development Model

กระบวนการพัฒนาของเราใช้แนวทาง Incremental + Iterative Development

Incremental คือการเพิ่มฟีเจอร์ทีละส่วน ส่วน Iterative คือการนำ feedback หรือปัญหาจาก phase ก่อนหน้ามาปรับปรุงใน phase ถัดไป

Phase 1 เราเก็บ Requirement, Scope และ FR/NFR
Phase 2 เพิ่มการออกแบบระบบ Use Case และ UI Prototype
Phase 3 พัฒนาเว็บไซต์ที่ใช้งานได้จริง พร้อม Frontend, Backend และ Unit Test
Phase 4 ปรับปรุง UI/UX แก้บั๊ก เพิ่ม Profiling, CI/CD และ Documentation


สไลด์หน้า 10: Phase Problems and Solutions

ในแต่ละ Phase ทีมมีปัญหาและวิธีแก้ต่างกันครับ

Phase 1 ปัญหาคือทีมยังเข้าใจ flow ของระบบไม่ตรงกัน จึงแก้โดยทำ Prototype เพื่อให้เห็นภาพเดียวกัน

Phase 2 ปัญหาคือยังไม่คุ้นกับ Figma และ workflow บางส่วน จึงเพิ่มการสื่อสารในทีมและปรับ design ให้เข้ากับการพัฒนาเว็บจริง

Phase 3 พบปัญหาเชิงเทคนิค เช่น trip ข้ามเที่ยงคืน data ไม่ persist, SerpAPI quota และ merge conflict จึงเริ่มจัดการ bug แบ่ง branch งานให้ชัดขึ้น และเพิ่ม testing

Phase 4 ใช้ bug tracking, profiling, CI/CD และ structured error/fallback เพื่อทำให้ระบบเสถียรมากขึ้นครับ

สไลด์หน้า 11: Unit Testing

ส่วนต่อไปคือการทดสอบระบบครับ

ในฝั่ง Backend เราใช้ Jest ในการทำ Unit Testing เพื่อทดสอบ Business Logic หลักของระบบ โดยเฉพาะ class ทั้ง 3 ตัว คือ Place, SearchQuery และ TripPlanner

ตัวอย่างสิ่งที่ทดสอบ ได้แก่ การตรวจสอบพิกัด Latitude และ Longitude, การคำนวณระยะทางด้วย Haversine Formula, การแปลงหมวดหมู่สถานที่จาก API และการคำนวณเวลาหรือตรวจจับเวลาทับซ้อน

ผลการทดสอบล่าสุด Unit Test ผ่านทั้งหมด 65 tests จาก 3 test suites และมี statement coverage ประมาณ 97.33% ครับ


สไลด์หน้า 12: UI End-to-End Testing

ในส่วนของ UI Testing เราใช้ Playwright เพื่อจำลองการใช้งานเหมือนผู้ใช้จริง

มี test case หลัก 5 เคส ได้แก่ การกรอกพิกัดและกดสแกน การบันทึก Favorites และ refresh แล้วข้อมูลไม่หาย การตรวจเวลาทับซ้อนใน Trip Planner การตรวจ input validation และการเปลี่ยนภาษา/Theme

ผลล่าสุด UI E2E Test ผ่านทั้งหมด 5 จาก 5 test cases ครับ


สไลด์หน้า 13: Static Profiling

สำหรับ Static Profiling เราใช้ ESLint เพื่อตรวจสอบคุณภาพโค้ดโดยไม่ต้องรันโปรแกรม

ผลที่พบมี warning เรื่อง complexity ในบางจุด เช่น backend function normalizeServerPlace และ frontend function createPlaceCard เพราะมีการจัดการข้อมูลและ UI ที่ซับซ้อนขึ้น

อย่างไรก็ตาม ไม่พบ error หรือ critical bug และจำนวน no-unused-vars ลดลง แสดงว่าโครงสร้างโค้ดโดยรวมดีขึ้นและยังอยู่ในเกณฑ์เสถียรครับ


สไลด์หน้า 14: Dynamic Profiling

ในส่วน Dynamic Profiling เราใช้ Chrome DevTools และ Lighthouse เพื่อวัดประสิทธิภาพขณะใช้งานจริง

ผลที่ได้คือ Lighthouse score อยู่ที่ 87, Rendering Time อยู่ที่ 35 milliseconds และ Total Blocking Time หรือ TBT อยู่ที่ 0 milliseconds

ตัวเลขนี้แสดงว่าหน้าเว็บตอบสนองได้ดี UI ไม่ค้าง และมี performance ที่เหมาะสมสำหรับการใช้งานทั่วไปครับ


สไลด์หน้า 15: CI/CD Pipeline

ในส่วน CI/CD เราใช้ GitHub Actions เป็น CI Pipeline เพื่อรัน automated tests ทุกครั้งที่มีการ push code

Workflow หลักคือ เมื่อมีการ push ระบบจะดึงโค้ดล่าสุดขึ้น runner จากนั้นรัน ESLint เพื่อตรวจโครงสร้างโค้ด รัน Jest Unit Tests และรัน Playwright UI Tests เพื่อจำลองการเปิด browser แบบอัตโนมัติ

ถ้าทุกขั้นตอนผ่าน ระบบจะสรุปรายงานออกมา ทำให้ทีมมั่นใจได้ว่าโค้ดที่รวมเข้า branch หลักยังไม่ทำให้ระบบหลักพัง

หมายเหตุ: ตอนนี้เป็น CI Pipeline เป็นหลัก ส่วน CD หรือ deployment จริงยังเป็นส่วนที่สามารถพัฒนาต่อในอนาคตครับ


สไลด์หน้า 16: Member

สำหรับสมาชิกในกลุ่มมี 3 คนครับ

คนแรกคือ นายสิทธิโชติ เกรียงชัยพฤกษ์
คนที่สองคือ บุญยศักดิ์ รัตนดิลก ณ ภูเก็ต
และคนที่สามคือ นายอนันฌานนท์ แป้นสุวรรณ

โดยงานถูกแบ่งกันตามส่วนหลัก เช่น Frontend และ UI, Backend และ API, Testing, CI Pipeline และ Documentation ครับ


สไลด์หน้า 17: Thank You

โดยสรุป PinMe เป็นเว็บที่ช่วยให้ผู้ใช้ค้นหาสถานที่ใกล้เคียง บันทึกสถานที่ที่สนใจ และวางแผนทริป 1 วันได้ในระบบเดียว

ระบบมี 3 หน้า มีการเก็บข้อมูลสำคัญใน localStorage มีการเชื่อมต่อ SerpAPI ผ่าน Backend และมีการทดสอบทั้ง Unit Test, UI E2E Test, Profiling และ CI Pipeline เพื่อยืนยันคุณภาพของระบบ
